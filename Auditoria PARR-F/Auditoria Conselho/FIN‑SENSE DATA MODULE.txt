Documento técnico: Módulo de Banco de Dados — FIN‑SENSE DATA MODULE (versão 1.0 — 03/04/2026)

Objetivo

Armazenar, versionar e expor exclusivamente dados numéricos e metadados associados (sem cálculos/analytics embutidos). Serve como a camada persistente para que outros módulos (research, execution, UI) consumam dados para análises, backtests, correlações e métricas.
Escopo

Tabelas primárias para market data, orders/executions, orderbook snapshots, securities master, positions, benchmarks, corporate actions, rates, macro events, derivatives chain, provenance, QC flags, system metrics e counterparty exposures.
Contratos de ingestão, DDL Parquet/Delta, exemplos JSON de ingest, políticas de particionamento, chaves, versionamento e APIs de consulta.
Não inclui funções de cálculo, agregação ou modelos: apenas armazenamento, integridade, lineage e exposição.
Design principes

Imutabilidade de ingestões raw (append‑only). Derivados podem ser materializados em tabelas curated com upserts via Delta/Iceberg.
Todos timestamps UTC ISO8601 (ms). Registrar source_ts (origem) e recv_ts (chegada).
Metadados obrigatórios por linha: ingestion_id, source, schema_version, file_hash (opcional por batch), row_checksum, recv_ts, source_ts, operator_id, confidence_score, qc_flags.
Identificadores padronizados: FIGI preferencial + ISIN + ticker+exchange composite key. Rejeitar ou sinalizar linhas sem identificador crítico.
Particionamento principal por date (ingest_date / event_date conforme tabela) e clustering/indices por symbol/pair/portfolio_id.
Plataforma recomendada: object storage + Delta Lake (ou Iceberg) on compute layer (Spark/Trino), Kafka for streaming ingest, catalog (Glue/Metastore), schema registry (Avro/Protobuf).

Particionamento, clustering e índices

Particionar por date (UTC) usando event/ingest date depending on use-case:
Market ticks/orderbook: partition by event_date (date(timestamp_utc))
Orders/Execs/Positions: partition by send_date/exec_date/position_date
Clustering/Sort keys: symbol, exec_ts, order_id — otimiza range scans and joins.
Small tables (securities_master, provenance) not partitioned but partition by country or year if large.
Z‑ordering (Databricks) or sort optimization for common predicate columns (symbol, ingestion_id).
Versionamento e time travel

Use Delta Lake or Iceberg for ACID upserts and time travel.
Record schema_version and batch_ingest_id in each batch.
Maintain immutable daily manifests with:
manifest_id, ingest_id, file_list, file_hashes, total_rows, operator_id, signature, manifest_ts.
Enable time travel for curated tables (retention period policy e.g., 30–90 days depending on infra).
Contratos de ingestão (idempotência e deduplicação)

Each batch MUST include ingest_id (UUID), source, schema_version and manifest.
Idempotency: replays with same ingest_id rejected/ignored.
Dedup keys:
Ticks/raw trades: (source, seq_no) or trade_id.
Orders: order_id.
Executions: exec_id.
Validation on ingest:
Reject rows missing critical identifier (figi/isin/symbol) or set qc_flag=CRIT_MISSING_ID and route to quarantine.
Row checksum verification if provided.
Example payload (JSON minimal): { "ingest_id":"uuid", "source":"providerA", "schema_version":"v1.2", "records":[ { "symbol":"AAPL.N", "figi":"BBG000B9XRY4", "timestamp_utc":"2026-04-03T12:34:56.123Z", "seq_no":12345, "bid":173.45, "ask":173.47, "trade_price":173.46, "trade_size":100, "trade_id":"pA-123456", "recv_ts":"2026-04-03T12:34:56.130Z" } ] }
Checks automáticos mínimos (aplicáveis no ingest pipeline; sem cálculos de negócio)

Row‑level:
Presence of critical IDs (figi/isin or symbol+exchange) — severity Alta.
Timestamp sanity: source_ts <= recv_ts + allowed_skew? and not future > N minutes — flag/generate QC.
Price bounds: price > 0 and within exchange bounds (if available) or within extreme quantile thresholds — flag.
Duplicate detection via dedup keys.
Row checksum verification (if provided).
Batch‑level:
Row count within expected range vs historical mean ±σ — flag medium.
File checksum/manifest signature verification — fail on mismatch.
Volume reconciliation (sum sizes) vs exchange daily report (if available) — flag high.
Cross‑source:
Price divergence threshold (bps) between primary and secondary provider for same timestamp — flag high.
QC flags persisted in TBL_DATA_QUALITY_FLAGS and row qc_flags field in primary table.
Lineage e metadados

For every derived/curated row store:
source_ingest_ids (array), code_version (git hash), transform_job_id, transform_ts.
All outputs must reference ingest_id(s) used for derivation and manifest_id.
APIs de consulta e padrões de acesso

Exposição via:
SQL access (Trino/Spark SQL/Databricks SQL) com catalog.
REST endpoints (read-only) com pagination and query filters (symbol, start_ts, end_ts, ingestion_id, schema_version).
Parquet/Delta export endpoints for bulk pulls.
Exemplos de endpoints (contrato):
GET /v1/market_ticks?symbol=AAPL.N&start=2026-04-01T00:00:00Z&end=2026-04-02T00:00:00Z&ingest_id=uuid&page=1
GET /v1/orders/{order_id}
GET /v1/executions?order_id=...&start=...
Query patterns recomendados (sem cálculos):
Time range scans on partitioned date + symbol filter.
Use ingestion_id or schema_version to reproduce experiments.
Access control: RBAC, attribute‑based policies per sensitivity_tag.
Joins primários (contratos; sem agregações)

Exec → Orders: join on order_id (use indexed columns). Return exec rows with orders metadata and ingestion_ids for each side.
Market_Ticks → Orderbook_Snapshot: nearest snapshot join by timestamp (floor/ceil); return both ingestion_ids.
Positions_History → latest Market_Ticks: join on symbol with latest tick before valuation_ts using ingestion_id lineage.
Securities_Master join to any market table via figi or symbol+exchange.
Schema evolution e compatibilidade retroativa

Backwards compatible changes allowed: add nullable fields, add columns with defaults.
Breaking changes require schema_version bump and manifest noting migration steps.
Schema registry mandatory: Avro/Protobuf definitions with validators.
Retenção, compaction e tiers de armazenamento

Hot tier: raw ticks, orders, executions, last 1 year (fast access).
Warm tier: 1–7 years aggregated or 1m/5m derived (lower-cost compute).
Cold archive: 7+ years in compressed Parquet/CSV in cheaper object storage (Glacier/Archive).
Compaction: small files compaction daily/weekly to optimize read performance.
Segurança, governança e auditoria

Encryption in transit (TLS 1.2+) and at rest (AES‑256).
RBAC with least privilege. Sensitivity tags on tables/rows. MFA + strong auth for admin.
Audit logs for data access and schema changes; immutable manifests anchored by signature.
Masking/anonymization rules for dev/test environments; synthetic data for R&D.
Retention policy and legal hold support for regulatory requests.
Performance e dimensionamento

Streaming ingest with Kafka → transient topic → micro‑batch into Delta.
Use partition pruning + vectorized reads; provide materialized views for common access patterns (precomputed partitions only; no analytics inside module).
Scale compute independent of storage (elastic clusters) for bulk exports.

Exemplos de queries de join (sem cálculos)

Exec → Orders (SQL pattern) SELECT e.*, o.trader_id, o.strategy_id, o.symbol, e.ingestion_id AS exec_ingest, o.ingestion_id AS order_ingest FROM finsense.executions e JOIN finsense.orders o ON e.order_id = o.order_id WHERE e.exec_ts BETWEEN timestamp '2026-04-01 00:00:00' AND timestamp '2026-04-02 00:00:00' AND o.symbol = 'AAPL.N';
Market tick → nearest orderbook snapshot (pattern using window; returns both ingest ids) WITH snaps AS ( SELECT snapshot_id, ingestion_id AS snap_ingest, symbol, timestamp_utc AS snap_ts FROM finsense.orderbook_snapshot WHERE symbol='AAPL.N' AND snap_ts BETWEEN ... ) SELECT t.*, s.snap_ts, s.snap_ingest FROM finsense.market_ticks_raw t LEFT JOIN snaps s ON t.symbol = s.symbol AND s.snap_ts = (SELECT max(s2.snap_ts) FROM snaps s2 WHERE s2.snap_ts <= t.timestamp_utc) WHERE t.timestamp_utc BETWEEN ...; (Implementação prática: nearest timestamp join via range join/lead/lag or specialized time series join function in query engine.)
Catalogação de ingest/experiments (reprodutibilidade)

Every research job must record: input_ingest_ids[], query_sql, model_code_version, compute_job_id, output_table, job_ts. Store in a metadata registry table for reproducibility.
Quality SLA e alerting

SLA checks run per batch:
Ingest success/failure alert within 5 mins.
QC high severity alerts immediate (pager/email).
Daily report: rows ingested per source, QC summary, partition health.
Monitoring: Prometheus/Grafana metrics for ingestion throughput, lag, failed jobs, small files count.
